## :white_check_mark: Task Breakdown (실행 계획)
- [ ] **1. RecommendationCard 모델 스키마 작성** — `prisma/schema.prisma`에 추가:
  ```prisma
  model RecommendationCard {
    id               String    @id @default(cuid())
    userId           String
    ticker           String                        // 종목 심볼 (예: AAPL, TSLA)
    direction        String                        // 'BUY' | 'SELL' (Zod 검증)

    // 진입가: 단일가 또는 범위 중 택 1 (Zod .refine() 검증 - DTO-010)
    entryPrice       Float?                        // 진입 기준가 (단일 가격)
    entryRangeLow    Float?                        // 진입 범위 하한
    entryRangeHigh   Float?                        // 진입 범위 상한

    // 청산가: 단일가 또는 범위 중 택 1 (Zod .refine() 검증 - DTO-010)
    targetPrice      Float?                        // 청산 기준가 (단일 가격)
    targetRangeLow   Float?                        // 청산 범위 하한
    targetRangeHigh  Float?                        // 청산 범위 상한

    stopPrice        Float?                        // 손실 한도 기준가 (선택)
    holdDays         Int                           // 권장 보유 기간 1~10 정수 (Zod 검증)
    confidenceScore  String                        // 'aggressive' | 'balanced' | 'conservative' (Zod 검증)
    reasonLine       String                        // 한 줄 이유, 1~160자 (Zod .min(1).max(160))
    status           String                        // 'published' | 'no_call' | 'validation_failed' (Zod 검증)
    createdAt        DateTime  @default(now())
    validUntil       DateTime                      // 카드 유효 기한

    // Relations
    user             User      @relation(fields: [userId], references: [id])
    evidenceSnapshot EvidenceSnapshot?
    performanceRecord PerformanceRecord?

    @@index([userId, createdAt])
    @@index([ticker])
    @@index([status, createdAt])
  }
  ```

## :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: RecommendationCard 테이블 생성 성공**
- Given: `prisma/schema.prisma`에 RecommendationCard 모델이 정의된 상태
- When: `npx prisma db push`를 실행하면
- Then: 로컬 SQLite DB에 `RecommendationCard` 테이블이 생성되고, 18개 컬럼(id, userId, ticker, direction, entryPrice, entryRangeLow, entryRangeHigh, targetPrice, targetRangeLow, targetRangeHigh, stopPrice, holdDays, confidenceScore, reasonLine, status, createdAt, validUntil, userId FK)이 존재해야 한다.

**Scenario 2: 정상 카드(published) 저장**
- Given: User가 존재하는 상태
- When: 다음 데이터로 `prisma.recommendationCard.create()`를 실행:
  ```json
  {
    "userId": "<valid_user_id>",
    "ticker": "AAPL",
    "direction": "BUY",
    "entryPrice": 185.50,
    "targetPrice": 195.00,
    "holdDays": 5,
    "confidenceScore": "balanced",
    "reasonLine": "애플 WWDC 발표 이후 단기 상승 모멘텀 예상",
    "status": "published",
    "validUntil": "2026-04-25T00:00:00Z"
  }
  ```
- Then: 레코드가 정상 저장되고 cuid `id`가 자동 배정되어야 한다.

**Scenario 3: Range 기반 카드 저장**
- Given: User가 존재하는 상태
- When: `entryPrice` 대신 `entryRangeLow: 180.0`, `entryRangeHigh: 186.0`, `targetRangeLow: 192.0`, `targetRangeHigh: 198.0`으로 카드를 생성하면
- Then: 레코드가 정상 저장되고 `entryPrice`와 `targetPrice`는 null이어야 한다.

**Scenario 4: No Call 카드 저장**
- Given: LLM 호출이 실패한 상황
- When: `status: 'no_call'`, `reasonLine: '데이터 부족으로 추천을 생성할 수 없습니다'`로 카드를 생성하면
- Then: 레코드가 정상 저장되어야 한다.

**Scenario 5: User 관계 무결성**
- Given: RecommendationCard 테이블이 존재하는 상태
- When: 존재하지 않는 userId로 카드를 생성 시도하면
- Then: Prisma에서 Foreign key constraint 에러가 발생해야 한다.

**Scenario 6: 사용자별 오늘 카드 조회 (인덱스 활용)**
- Given: 특정 userId로 오늘 날짜에 3개 카드가 published 상태로 존재하는 상태
- When: `prisma.recommendationCard.findMany({ where: { userId, status: 'published', createdAt: { gte: todayStart } }, orderBy: { createdAt: 'desc' } })`를 실행하면
- Then: 3개 카드가 최신순으로 반환되어야 한다.

## :gear: Technical & Non-Functional Constraints
- **PK:** `cuid()` — SRS §6.2 규칙
- **ENUM 회피:** direction, confidenceScore, status 모두 String + Zod 검증 — SQLite 호환
- **가격 택 1 규칙:** entryPrice 또는 (entryRangeLow AND entryRangeHigh) — Prisma 레벨 검증 불가, **DTO-010에서 Zod `.refine()` 필수**
- **holdDays:** 1~10 정수 — Prisma 레벨 Int 타입, 범위는 **DTO-003 Zod 검증**
- **reasonLine:** 1~160자 — Prisma 레벨 String, 길이는 **DTO-003 Zod 검증**
- **3벌 카드:** 동일 userId/날짜에 aggressive/balanced/conservative 3개 카드 동시 저장 — REQ-FUNC-082
- **성능:** userId+createdAt 복합 인덱스 → 홈 카드 조회 p95 ≤ 800ms 기여 — REQ-NF-001
