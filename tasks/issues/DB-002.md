## :white_check_mark: Task Breakdown (실행 계획)
- [ ] **1. User 모델 스키마 작성** — `prisma/schema.prisma`에 아래 모델 추가:
  ```prisma
  model User {
    id            String    @id @default(cuid())
    email         String?   @unique
    name          String?
    signupChannel String    // 'email' | 'google' | 'kakao' (Zod 검증)
    timezone      String    @default("Asia/Seoul")
    consentPush   Boolean   @default(false)
    createdAt     DateTime  @default(now())
    updatedAt     DateTime  @updatedAt

    // Relations (이후 태스크에서 추가)
    riskProfile       RiskProfile?
    watchlist         Watchlist[]
    recommendationCards RecommendationCard[]

    // NextAuth.js 관계 (DB-008에서 추가)
    // accounts       Account[]
    // sessions       Session[]
  }
  ```
- [ ] **2. 필드별 제약 조건 확인 및 주석 추가**
  | 필드 | 타입 | 제약 | SRS 근거 |
  |---|---|---|---|
  | `id` | String | `@id @default(cuid())` | §6.2 PK 규칙 |
  | `email` | String? | `@unique`, NULLABLE | §6.2.1 — NextAuth.js 연동, 소셜 로그인 시 email 미제공 가능 |
  | `name` | String? | NULLABLE | §6.2.1 — 표시명, 선택 입력 |
  | `signupChannel` | String | NOT NULL | §6.2.1 — 'email'/'google'/'kakao', Zod로 검증 |
  | `timezone` | String | NOT NULL, DEFAULT 'Asia/Seoul' | §6.2.1 — 푸시 발송 시간대 기준 |
  | `consentPush` | Boolean | NOT NULL, DEFAULT false | §6.2.1, REQ-FUNC-052 — 푸시 수신 동의 |
  | `createdAt` | DateTime | `@default(now())` | §6.2.1 — 가입 일시 |
  | `updatedAt` | DateTime | `@updatedAt` | §6.2.1 — 최종 수정 일시 |

- [ ] **3. Relation 필드 placeholder 주석 배치** — 아직 생성되지 않은 모델(RiskProfile, Watchlist, RecommendationCard)에 대한 relation 필드는 주석으로 남기고, 해당 모델 태스크(DB-003~005)에서 양방향 관계를 완성

- [ ] **6. signupChannel Zod 검증 스키마 초안 작성** — `src/lib/schemas/user.ts` (DTO-* 태스크에서 확장):

## :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: User 테이블 생성 성공**
- Given: `prisma/schema.prisma`에 User 모델이 정의된 상태
- When: `npx prisma db push`를 실행하면
- Then: 로컬 SQLite DB에 `User` 테이블이 생성되고, 모든 컬럼(id, email, name, signupChannel, timezone, consentPush, createdAt, updatedAt)이 존재해야 한다.

**Scenario 2: cuid() PK 자동 생성**
- Given: User 테이블이 존재하는 상태
- When: Prisma Client로 `prisma.user.create({ data: { signupChannel: 'google' } })`를 실행하면
- Then: `id` 필드에 cuid 형식의 고유 문자열이 자동 할당되어야 한다.

**Scenario 3: email unique 제약 동작**
- Given: email이 `test@example.com`인 User가 이미 존재하는 상태
- When: 동일한 email로 `prisma.user.create()`를 시도하면
- Then: Prisma에서 Unique constraint 위반 에러(`P2002`)가 발생해야 한다.

**Scenario 4: email nullable 허용**
- Given: User 테이블이 존재하는 상태
- When: email 없이 `prisma.user.create({ data: { signupChannel: 'kakao' } })`를 실행하면
- Then: email이 null인 User 레코드가 정상 생성되어야 한다.

**Scenario 5: consentPush 기본값**
- Given: User 테이블이 존재하는 상태
- When: consentPush를 명시하지 않고 User를 생성하면
- Then: `consentPush` 필드의 값이 `false`이어야 한다.

**Scenario 6: signupChannel Zod 검증**
- Given: signupChannelSchema Zod 스키마가 정의된 상태
- When: `signupChannelSchema.safeParse('invalid_channel')`을 실행하면
- Then: `success: false`이고 에러 메시지가 반환되어야 한다.

## :gear: Technical & Non-Functional Constraints
- **PK 전략:** `cuid()` — SRS §6.2 규칙, SQLite/PostgreSQL 양쪽 호환
- **ENUM 회피:** signupChannel은 String + Zod 검증 — SQLite ENUM 미지원 대응 (SRS §6.2)
- **email NULLABLE:** 소셜 로그인(Google/Kakao) 시 email이 제공되지 않을 수 있음 — NextAuth.js 호환
- **timezone 기본값:** 'Asia/Seoul' — SRS §6.2.1, 대한민국 사용자 기본 타겟
- **개인정보 최소 수집:** email, name, signupChannel, timezone, consentPush만 저장 — SRS REQ-NF-024
- **데이터 삭제/익명화:** 탈퇴 시 User 데이터 처리 가능해야 함 — SRS REQ-FUNC-093 (AUTH-C04에서 구현)
