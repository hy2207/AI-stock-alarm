## :white_check_mark: Task Breakdown (실행 계획)
- [ ] **1. Prisma 패키지 설치** — `npm install prisma @prisma/client` 및 `npm install -D prisma`
- [ ] **2. Prisma 초기화** — `npx prisma init` 실행, `prisma/schema.prisma` 파일 생성 확인
- [ ] **3. 로컬 개발용 SQLite 데이터소스 설정**
  ```prisma
  // prisma/schema.prisma
  datasource db {
    provider = "sqlite"
    url      = env("DATABASE_URL") // 로컬: file:./dev.db
  }

  generator client {
    provider = "prisma-client-js"
  }
  ```
- [ ] **4. 환경 변수 설정** — `.env.local`에 SQLite 경로 설정:
  ```
  DATABASE_URL="file:./dev.db"
  ```
- [ ] **5. Prisma Client 싱글턴 유틸 작성** — `src/lib/db/prisma.ts`:
  ```typescript
  import { PrismaClient } from '@prisma/client';

  const globalForPrisma = globalThis as unknown as {
    prisma: PrismaClient | undefined;
  };

  export const prisma =
    globalForPrisma.prisma ??
    new PrismaClient({
      log: process.env.NODE_ENV === 'development' ? ['query', 'error', 'warn'] : ['error'],
    });

  if (process.env.NODE_ENV !== 'production') {
    globalForPrisma.prisma = prisma;
  }
  ```
  > **설계 의도:** Vercel Serverless 환경에서 매 요청마다 PrismaClient가 재생성되는 것을 방지하기 위한 global 싱글턴 패턴. Hot reload 시 연결 고갈 문제(REQ-NF-072) 대응.
- [ ] **6. Supabase(PostgreSQL) 전환 가이드 주석 추가** — `schema.prisma` 상단에 배포 환경 전환 방법 주석:
  ```prisma
  // ------------------------------------------------------------------
  // 배포 환경(Supabase PostgreSQL) 전환 시:
  //   1. provider = "postgresql"
  //   2. DATABASE_URL = Supabase Connection Pooler URL
  //   3. directUrl = Supabase Direct URL (마이그레이션용)
  //   4. npx prisma migrate deploy
  // ------------------------------------------------------------------
  ```
- [ ] **7. `.gitignore` 업데이트** — `prisma/*.db`, `prisma/*.db-journal` 추가
- [ ] **8. 빈 스키마 상태에서 Prisma Client 생성 확인** — `npx prisma generate` 실행 후 에러 없음 확인
- [ ] **9. SQLite ↔ PostgreSQL 호환성 제약 문서화** — `prisma/README.md`에 다음 호환성 규칙 기술:
  - PK: `String @id @default(cuid())` 사용 (UUID 대체, SQLite 호환)
  - ENUM: `String` 타입 + Zod 코드 레벨 검증 (SQLite ENUM 미지원 대응)
  - JSONB: `Json?` 타입 사용 (Prisma가 SQLite에서는 TEXT, PostgreSQL에서는 JSONB로 자동 매핑)
  - DateTime: `@default(now())`, `@updatedAt` 사용
  - 비즈니스 규칙: Server Action/Route Handler에서 Zod 스키마로 검증

## :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: Prisma Client 정상 생성**
- Given: `prisma/schema.prisma`에 datasource가 SQLite로 설정된 상태
- When: `npx prisma generate`를 실행하면
- Then: `node_modules/.prisma/client/`에 타입 안전 클라이언트가 에러 없이 생성되어야 한다.

**Scenario 2: 로컬 SQLite DB 파일 생성**
- Given: `DATABASE_URL="file:./dev.db"`가 `.env.local`에 설정된 상태
- When: 임의의 모델을 추가하고 `npx prisma db push`를 실행하면
- Then: `prisma/dev.db` 파일이 생성되고, 테이블이 정상 반영되어야 한다.

**Scenario 3: Prisma Client 싱글턴 동작**
- Given: `src/lib/db/prisma.ts`가 작성된 상태
- When: 개발 서버에서 여러 API 요청이 발생하면
- Then: PrismaClient 인스턴스가 1개만 유지되고, 연결 풀 경고가 발생하지 않아야 한다.

**Scenario 4: Git 추적 제외 확인**
- Given: `.gitignore`에 `prisma/*.db` 규칙이 추가된 상태
- When: `prisma/dev.db` 파일이 생성된 후 `git status`를 확인하면
- Then: `dev.db` 파일이 Untracked/Modified 목록에 나타나지 않아야 한다.

**Scenario 5: PostgreSQL 전환 준비 상태**
- Given: SQLite 기반 스키마와 Prisma Client가 동작하는 상태
- When: `schema.prisma`의 provider를 `"postgresql"`로 변경하고 DATABASE_URL을 Supabase URL로 설정하면
- Then: `npx prisma generate`가 에러 없이 성공해야 한다. (실제 Supabase 연결은 INFRA-004에서 검증)

## :gear: Technical & Non-Functional Constraints
- **ORM:** Prisma ORM 필수 — SRS C-TEC-003
- **로컬 DB:** SQLite — 인프라 설정 복잡도 최소화
- **배포 DB:** Supabase (PostgreSQL) — SRS C-TEC-003
- **PK 전략:** `cuid()` 사용 — UUID 대신 SQLite 호환 고유 식별자 (SRS §6.2)
- **ENUM 대체:** String + Zod 검증 — SQLite ENUM 미지원 대응 (SRS §6.2)
- **연결 관리:** globalThis 싱글턴 패턴 — Serverless 환경 연결 풀 고갈 방지 (SRS REQ-NF-072)
- **보안:** 데이터베이스 파일(*.db)은 Git 추적에서 제외 — 민감 데이터 유출 방지
